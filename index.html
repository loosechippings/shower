Background

  1. nn business objects
  2. Complexity of nn

  4. nn lines of code
  5. 12+ releases a year
  6. Built on RTTP

Complexity measured as number of paths through the code.

Problems

All domain objects are public which leads to high coupling of services to domain objects. There is high complexity both functionally and technically. High coupling and high complexity leads to a need for end-to-end testing and long release cycles. The RTTP framework requires specialist knowledge to develop on. The application is not cloud native in that it parts of it are not easy to scale, it is hard to make resiliant to failure.

Features of RTTP

  • Asynchronous persistence
  • Publish/subscribe mechanism
  • Query by example
  • Load balancing
  • Service monitoring

Challenges of Removing RTTP

  • Business logic and framework API calls are mixed

Principals

  • Favour an event sourcing model over mutating state
  • Favour industry standards over internal models
  • Favour language agnostic communiction

Recommendation

Take a well defined, isolated function of the system - e.g. the Booking Integrity Control.
